Why does Docker exist?

Imagine you build a Django app on your laptop. It works perfectly. Then your friend tries to run it on their machine — and it crashes. Why? Because they have a different Python version, different libraries, different operating system settings.

This problem is so common it even has a name: "it works on my machine".

🍱 Real-world analogy Think of Docker like a lunchbox. Your food (code), the plate (Python), the utensils (libraries), and even the sauce (config) are all packed inside together. No matter whose table you put it on — your laptop, a server, a friend's computer — it opens and works exactly the same.

Docker solves this by packaging your entire app + its environment into a single, portable unit that runs identically everywhere.

The Four Core Concepts

🖼️

Docker Image

A blueprint of your project. It contains your code, the OS layer, Python, all your libraries — everything needed to run. Like a recipe or a class definition.

Read-only. Reusable. Shareable. Never changes once built.

blueprint / template
📦

Docker Container

A running instance of an image. Like creating an object from a class. You can start it, stop it, restart it, or delete it easily — and create multiple from the same image.

Lightweight, isolated, disposable.

running instance
💾

Volumes

By default, when a container stops or restarts, all data inside it is wiped. Volumes connect a folder inside the container to a real folder on your computer so data survives restarts.

Think of it like a shared USB drive between container and your PC.

persistent storage
🏪

Docker Hub

The App Store of Docker images. Thousands of pre-built images are hosted here — official Redis, PostgreSQL, Python, Nginx, and more. You pull them instead of building from scratch.

Free account at hub.docker.com to push/pull images.

image registry
🎯 Quick mental model Image = Class definition. Container = Object instance. Dockerfile = the code that writes the class. Docker Hub = GitHub for images.

The Dockerfile — Building Your Image

A Dockerfile is a text file with step-by-step instructions for building a Docker image. Docker reads it top to bottom and executes each instruction like a script.

Your Dockerfile is what makes your image unique — it says "start with official Python 3.11, copy my code in, install my dependencies, then start my app."

🏗️ Analogy: Building a house The Dockerfile is the architectural blueprint. Docker is the construction crew. The finished house is the Image. When someone moves in (runs the container), they live in a house built exactly from that blueprint.
Dockerfile (Django)
dockerfile 
# ── Step 1: Choose a base image from Docker Hub ──────────────────
# This gives us Python 3.11 on a minimal Linux (Alpine) OS
FROM python:3.11-slim

# ── Step 2: Set working directory inside the container ───────────
# All commands from here run inside /app
WORKDIR /app

# ── Step 3: Copy requirements first (smart caching trick) ────────
# If requirements.txt didn't change, Docker reuses the cached layer
COPY requirements.txt .

# ── Step 4: Install Python dependencies ──────────────────────────
# --no-cache-dir keeps image smaller
RUN pip install --no-cache-dir -r requirements.txt

# ── Step 5: Copy the rest of the project code ────────────────────
COPY . .

# ── Step 6: Expose the port your app listens on ──────────────────
# This is documentation; actual port binding happens with -p flag
EXPOSE 8000

# ── Step 7: The command to start the app ─────────────────────────
# CMD runs when the container starts. Only ONE CMD allowed.
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
⚠️ Django Dev vs Production runserver is for development only. In production, use gunicorn: replace CMD with ["gunicorn", "myproject.wsgi:application", "--bind", "0.0.0.0:8000"]. Also add gunicorn to requirements.txt.
✅ Django Tips Set ALLOWED_HOSTS = ['*'] in settings.py during development inside Docker. Also set DJANGO_SETTINGS_MODULE as an ENV variable in Dockerfile: ENV DJANGO_SETTINGS_MODULE=myproject.settings
Dockerfile (Node.js / Express)
dockerfile 
# Start with official Node.js image
FROM node:20-alpine

WORKDIR /app

# Copy package files first for caching
COPY package*.json ./

# Install dependencies
RUN npm install --production

# Copy source code
COPY . .

EXPOSE 3000

# Start the Express server
CMD ["node", "server.js"]
Dockerfile (Generic Python / Flask)
dockerfile 
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 5000
# Flask app entry point
CMD ["flask", "run", "--host=0.0.0.0"]

Building & Running Your Image

Terminal Commands
bash 
# Build an image from your Dockerfile
# -t gives it a name (tag). The "." means "Dockerfile is in current folder"
docker build -t my-django-app .

# Run the image as a container

Going Inside a Container

Every container is basically a mini Linux computer running in isolation. You can "SSH" into it using docker exec to inspect it, run Django management commands, check logs, or debug issues.

Container Interaction Commands
bash 
# ── Get a terminal (bash shell) inside a running container ───────
# -it = interactive + allocate a terminal
docker exec -it django-container bash

# Some minimal images only have sh (not bash)
docker exec -it django-container sh

# Run a single command without entering the container
docker exec django-container python manage.py migrate

# Run Django shell inside container
docker exec -it django-container python manage.py shell

# View live logs of a container (Ctrl+C to stop)
docker logs -f django-container

# View last 50 lines of logs
docker logs --tail 50 django-container

# Copy a file FROM container TO your machine
docker cp django-container:/app/db.sqlite3 ./db.sqlite3

# Copy a file FROM your machine INTO container
docker cp ./myfile.py django-container:/app/myfile.py

# Inspect container details (IP, env vars, mounts, etc.)
docker inspect django-container

Docker Compose — Handling Multiple Containers

Real apps are never just one service. A Django app needs: the web server, a database (PostgreSQL), maybe Redis for caching, maybe Celery for background jobs. Managing each with separate docker run commands becomes a nightmare.

Docker Compose is a tool that lets you define all your services in a single file (docker-compose.yml) and start everything with one command: docker compose up.

🎭 Analogy: Orchestra conductor Docker Compose is the conductor. Each instrument (container) can play alone, but the conductor starts them all in the right order, makes sure they can hear each other, and stops them together when the concert ends.

When is Docker Compose necessary?

1

Multi-service apps

Your Django app needs PostgreSQL + Redis. Instead of 3 separate docker run commands with custom networking, one docker-compose.yml handles it all.

2

Service dependencies & startup order

Redis must start before your Django app. Compose lets you say depends_on: redis so Docker starts services in the right order.

3

Shared networking automatically

All services in a Compose file are on the same virtual network. They can reach each other by service name (e.g., redis, db) without any IP configuration.

4

Reproducible team environments

Your whole team clones the repo, runs docker compose up, and has an identical environment in minutes. No more "works on my machine".

Dockerfile vs docker-compose.yml — What's the difference?

Real Example: Redis + Redis Insight

Your instructor likely used this example. Let's understand exactly what's happening, how the two services communicate, and how Redis Insight (a web-based GUI tool) talks to Redis inside Docker.

What are these two services?

🗄️

Redis

The actual database service. It stores your data in memory (super fast). Pulled from Docker Hub as redis:latest. Listens on port 6379 by default.

port 6379
🌐

Redis Insight

A beautiful web-based GUI to browse and manage your Redis data visually. Instead of typing commands, you click around in a browser. It connects to Redis to show you what's stored.

port 5540 (browser)

How do they communicate? (Critical concept)

Docker Network Architecture — Redis Example
Docker Network (internal)
redis (container name = DNS name)
redisinsight
⇄
Your Browser (outside)
localhost:5540 → Redis Insight UI
✗ Cannot reach redis directly
🔑 Key insight: Containers talk using service names as DNS hostnames. redisinsight connects to Redis using hostname = redis (the service name) and port 6379. Docker resolves redis → internal container IP automatically.

When you open Redis Insight in your browser at localhost:5540 and configure a new Redis connection, you set:

  • Host: redis — (not "localhost"! This is the service name inside Docker's network)
  • Port: 6379

Docker's built-in DNS server sees the hostname redis, looks up which container has that name, and converts it to the internal IP address. This is called container networking via service discovery.

⚠️ Common Beginner Mistake When Redis Insight asks for the Redis host, do NOT type localhost. Inside Docker's network, localhost means the Redis Insight container itself — not Redis! Use the service name redis instead.

The docker-compose.yml file for this setup

docker-compose.yml — Redis + Redis Insight
yaml 
# Docker Compose file version
version: '3.8'

services:

    # ── Service 1: Redis (the actual database) ─────────────────────
    redis:
        # Pull the official Redis image from Docker Hub
        image: redis:latest

        # Container name for easy identification
        container_name: redis

        # Map port 6379 on your machine to 6379 inside container
        # Format: "HOST_PORT:CONTAINER_PORT"
        ports:
            - "6379:6379"

        # Persist Redis data even if container restarts
        # ./redis-data on your machine ↔ /data inside container
        volumes:
            - ./redis-data:/data

        # Restart automatically if it crashes
        restart: unless-stopped

    # ── Service 2: Redis Insight (the web GUI) ─────────────────────
    redisinsight:
        image: redislabs/redisinsight:latest
        container_name: redisinsight

        # Access the UI at http://localhost:5540
        ports:
            - "5540:5540"

        # Start redisinsight ONLY AFTER redis is running
        # This ensures Redis is ready before the GUI tries to connect
        depends_on:
            - redis

        restart: unless-stopped

# All services automatically share a network.
# redisinsight can reach redis using hostname "redis" (service name)
▶️ How to start everything Run docker compose up -d in the folder with this file. Then open http://localhost:5540 in your browser. In Redis Insight, add a database connection with host redis and port 6379.

Full Django + Redis + PostgreSQL Example

docker-compose.yml — Django Full Stack
yaml 
version: '3.8'

services:

    # ── PostgreSQL Database ─────────────────────────────────────────
    db:
        image: postgres:15
        container_name: postgres_db
        environment:
            # These become environment variables inside the container
            POSTGRES_DB: myproject
            POSTGRES_USER: admin
            POSTGRES_PASSWORD: secret123
        volumes:
            - postgres_data:/var/lib/postgresql/data
        ports:
            - "5432:5432"

    # ── Redis Cache ─────────────────────────────────────────────────
    redis:
        image: redis:7-alpine
        container_name: redis
        ports:
            - "6379:6379"

    # ── Django Web App ──────────────────────────────────────────────
    web:
        # Build from the Dockerfile in the current directory
        build: .
        container_name: django_web
        command: python manage.py runserver 0.0.0.0:8000
        volumes:
            # Mount local code into container (live reload during dev)
            - .:/app
        ports:
            - "8000:8000"
        environment:
            # Django connects to PostgreSQL using service name "db"
            DATABASE_URL: postgresql://admin:secret123@db:5432/myproject
            # Django connects to Redis using service name "redis"
            REDIS_URL: redis://redis:6379/0
            DEBUG: "True"
        depends_on:
            # Django waits for db and redis to be up before starting
            - db
            - redis

# Named volumes — managed by Docker, persists data
volumes:
    postgres_data:
🐍 Django settings.py configuration In your Django settings, use os.environ.get('DATABASE_URL') and os.environ.get('REDIS_URL') to read from the environment variables set in Compose. Use dj-database-url and django-redis packages.

Dockerfile Instructions Reference
FROM python:3.11-slim
Starting point. Base image to build upon. Always the first instruction.
WORKDIR /app
Working directory. All following commands run from this path inside the container.
COPY src dest
Copy files from your machine into the container image.
ADD src dest
Like COPY but also handles URLs and auto-extracts .tar archives. Prefer COPY for simplicity.
RUN pip install ...
Execute commands during image build. Creates a new layer. Use for installing packages.
ENV KEY=VALUE
Set environment variable. Available both during build and when container runs.
ARG BUILD_VAR
Build-time variable. Only available during docker build, not at runtime.
EXPOSE 8000
Documentation of which port the app uses. Does NOT actually open the port — use -p flag for that.
CMD ["python", "app.py"]
Default command when container starts. Can be overridden at runtime. Only one CMD.
ENTRYPOINT ["gunicorn"]
Like CMD but cannot be overridden. Often used for the main executable; CMD becomes its arguments.
VOLUME ["/data"]
Declare a mount point. Tells Docker this path should be stored as a volume.
USER appuser
Run as non-root user. Security best practice for production images.
HEALTHCHECK CMD ...
Define a command Docker uses to check if the container is healthy.
LABEL key="value"
Add metadata to image (author, version, description).

Frequently Asked Questions

Quick answer: -d does NOT exist for docker build — it's for docker run. This is a common confusion!

-t (with docker build): Tags your image with a name. Example: docker build -t my-app . builds an image called "my-app". Without -t, Docker gives it an ugly random ID like a3f2b... that's hard to remember.

-d (with docker run): Detached mode. Runs the container in the background so your terminal is free. Without -d, the container's output streams directly to your terminal and you can't do anything else until you Ctrl+C (which stops the container).

So the correct usage is: docker build -t my-app . then docker run -d -p 8000:8000 my-app

Absolutely necessary when: You have 2+ services that need to communicate (e.g., Django + PostgreSQL + Redis). You could technically start each with docker run and connect them manually with --network flags, but Compose makes it 10x simpler and reproducible.

Optional (but still useful) when: You have a single container app. You can use Compose just to store your run configuration as a file instead of remembering long docker run commands.

Concrete example — without Compose:

docker network create mynet → docker run -d --network mynet --name redis redis → docker run -d --network mynet -p 8000:8000 -e REDIS_URL=redis://redis:6379 my-django-app

With Compose: Write the YAML once, then just docker compose up. The same result, but readable and shareable.

Each Docker container has its own network namespace — it's like a separate mini-computer. So localhost inside Container A refers to Container A itself, not Container B.

Docker Compose automatically creates a shared virtual network for all services in a Compose file. On this network, each service gets a DNS entry equal to its service name. So if you name a service redis, Docker's internal DNS resolves redis to that container's IP address.

That's why in Redis Insight you type host = redis (not localhost): Redis Insight is running inside Docker too, and it uses Docker's DNS to find the Redis container by name.

Rule of thumb: Use localhost when YOUR BROWSER accesses a published port. Use the service name when one container talks to another.

This is the Volumes concept! Container filesystems are temporary — when a container is deleted or recreated, all internal data is gone. This is intentional (keeps containers stateless and reproducible).

Fix it with a volume mount:

In docker run: docker run -v ./my-data:/data redis

In docker-compose.yml: add volumes: - ./my-data:/data under the service.

This says: "The /data folder inside the container is actually stored at ./my-data on your machine." When the container restarts, it reads from that real folder, so data persists.

For databases, always add a volume. For code, you can also mount your source folder so changes reflect instantly without rebuilding.

An image is a static, read-only snapshot — like a frozen template. It never changes. You can create as many containers from the same image as you want.

A container is a running (or stopped) instance of that image. It's alive, it has its own state, files may be written to it. But those written files are lost if the container is removed (unless you use volumes).

Analogy: An image is like a cookie cutter. A container is each cookie you stamp out. All cookies look the same (from the same cutter), but you can decorate them individually, and eating one doesn't affect the cutter or the other cookies.

Once your containers are running, use docker compose exec to run management commands inside the web container:

docker compose exec web python manage.py migrate

docker compose exec web python manage.py createsuperuser

docker compose exec web python manage.py collectstatic

Replace web with whatever you named your Django service in the Compose file.

Pro tip: You can also add a startup script. Create a file entrypoint.sh that runs migrations automatically when the container starts, then set ENTRYPOINT ["./entrypoint.sh"] in your Dockerfile.

When you run docker build, Docker copies your code INTO the image at that moment. It's a snapshot. If you change your code afterward, the image still has the old version.

The fix for development: Mount your source code as a volume! In docker-compose.yml, add under your web service: volumes: - .:/app

This mounts your entire project folder into the container. Now changes to files on your machine are instantly reflected inside the container — no rebuild needed. Django's auto-reloader will pick up the changes automatically.

For production: You DO want code baked into the image (remove the volume mount) so the image is self-contained and deployable anywhere.

The format is HOST_PORT:CONTAINER_PORT — always left is your machine, right is inside the container.

Example: -p 8080:8000

Your machine port 8080 → maps to → Container port 8000

So you'd open http://localhost:8080 in your browser, and it sends requests into port 8000 inside the container where Django is listening.

Tip: If you want the same number on both sides, just use -p 8000:8000. You only need different numbers if port 8000 is already taken on your machine.

Think of it like: "My front door (8080) leads to their door (8000)".

A Virtual Machine includes a full operating system (Windows, Linux, etc.) simulated on top of your OS. It's heavy — takes gigabytes of RAM and minutes to start.

A Docker container shares your machine's OS kernel. It only packages the app and its dependencies, not a full OS. This makes containers start in seconds and use far less memory.

Think of it this way: a VM is like renting an entire apartment. A Docker container is like renting just one room in a shared house. Both give you private space, but the room is much lighter and cheaper.

Docker is not a replacement for VMs — they solve similar problems at different layers. VMs are better for full OS isolation; containers are better for app packaging and microservices.

Never hardcode secrets in your Dockerfile or docker-compose.yml — especially if you push code to GitHub.

Best practice: Create a .env file in your project root:

SECRET_KEY=your-secret-here
DEBUG=True
DATABASE_URL=postgresql://admin:pass@db:5432/mydb

In docker-compose.yml, reference it: env_file: - .env

In Django settings.py: import os; SECRET_KEY = os.environ.get('SECRET_KEY')

Add .env to .gitignore so it's never committed. Share an .env.example file with fake values as a template for teammates.

1. Run docker logs <container-name> — this shows the error message from your app. Most of the time, the problem is immediately obvious here.

2. If the container exits immediately, try running it without the -d flag so you see the output live: docker run -p 8000:8000 my-app
